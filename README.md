# 📚 多策略 RAG 文件問答系統

一個**單機優先 (Local-First)** 的文件問答工具，支援 PDF / DOCX 上傳，內建 **8 種 RAG 檢索策略**，
搭配 **Groq API** 進行問答生成，並可選擇性串接 **Supabase** 保存雲端問答紀錄。

不需要任何後端伺服器，也不需要資料庫管理員 — 每個使用者自己申請免費金鑰即可獨立運作，
所有金鑰皆可直接在網頁 UI 輸入，不需要碰觸任何設定檔。

---

## ✨ 功能特色

- 📄 支援上傳 **PDF / DOCX** 文件，自動切片與向量化
- 🧠 內建 **8 種 RAG 檢索策略**：相似度檢索、MMR、混合檢索、多查詢、上下文壓縮、父子文件、自我查詢、重排序
- 🗄️ 使用 **ChromaDB** 作為本機向量資料庫，不需外部服務即可運作
- 🔑 **BYOK（Bring Your Own Key）** 模式：每位使用者自行輸入自己的 Groq API Key，無需共用金鑰
- ☁️ 選配 **Supabase 雲端問答紀錄**：可直接在側邊欄輸入金鑰，或由部署者預先設定，問答歷史會自動同步保存，重新整理頁面也不會消失

---

## 🚀 快速開始

### 1. 安裝依賴套件

```bash
pip install -r requirements.txt

2. 啟動應用程式
streamlit run streamlit_app.py

3. 開始使用

打開網頁後，先在左側 Step 0 貼上你的 Groq API Key，點擊「套用金鑰」
（選填）展開左側 ☁️ Supabase 連線設定，貼上你的 Supabase 金鑰，啟用雲端問答紀錄
在 Step 1 上傳一份 PDF 或 DOCX 文件，點擊「建立向量索引」
在 Step 2 選擇一種 RAG 策略
在主畫面輸入問題，點擊「送出問題」即可獲得回答


🔑 申請必要金鑰
本系統需要一把必要金鑰（Groq）與一組選配金鑰（Supabase）。
必要：Groq API Key（用於問答生成）

前往 console.groq.com/keys
用 Google 或 GitHub 帳號登入（免費，不需信用卡）
點擊 Create API Key，複製產生的金鑰（以 gsk_ 開頭）
貼到系統左側 Step 0 的輸入框，點擊「套用金鑰」即可


免費額度足夠日常測試使用，每個人建議申請自己專屬的一把，不要共用。

選配：Supabase 金鑰（用於雲端問答紀錄，共 2 把）
如果你想讓問答紀錄在重新整理頁面後依然保留，可以申請這組金鑰。這組金鑰有兩種填入方式，任選一種即可：
| 方式 | 適合情境 |
|---|---|
| ① 直接在網頁 UI 輸入（推薦） | 每個使用者用自己的 Supabase 專案，紀錄互相獨立，不需修改任何檔案 |
| ② 寫入 .streamlit/secrets.toml | 部署者想讓所有訪客共用同一個資料庫，或想省去每次手動輸入的步驟 |
系統會優先讀取 UI 輸入的值，若 UI 留空，才會自動退回讀取 secrets.toml（如果有設定的話）。兩種都沒有設定也完全沒問題，系統會自動改用本機 Session 暫存問答紀錄，不影響核心功能。
步驟一：建立 Supabase 專案並取得金鑰

前往 supabase.com，用 GitHub 或 Email 註冊（免費）
點擊 New Project，填寫專案名稱、資料庫密碼、選擇區域（建議選 Northeast Asia (Tokyo)），建立專案
建置完成後，進入 Project Settings → API，找到以下兩個值：

| 名稱 | 樣式 | 用途 |
|---|---|---|
| Project URL | https://xxxxxxxx.supabase.co | 專案網址 |
| anon public key | 以 eyJ 開頭的一長串字串 | 存取金鑰 |
步驟二：建立資料表
到 Supabase 的 SQL Editor，執行以下指令建立問答紀錄表：
create table if not exists qa_history (
    id bigint generated by default as identity primary key,
    question text not null,
    answer text not null,
    strategy text,
    created_at timestamptz default now()
);

alter table qa_history enable row level security;

create policy "Allow public insert"
on qa_history for insert
to anon
with check (true);

create policy "Allow public select"
on qa_history for select
to anon
using (true);

步驟三：填入金鑰（選其中一種方式）
方式① UI 輸入（無需碰任何檔案）
打開網頁，展開左側「☁️ 點此輸入 Supabase 金鑰」，把 Project URL 和 anon public key 分別貼進兩個輸入框即可，系統會立即顯示「✓ Supabase 已連線」。
方式② 寫入 secrets.toml（部署者統一設定）
在專案根目錄建立 .streamlit/secrets.toml：
[supabase]
url = "貼上你的 Project URL"
key = "貼上你的 anon public key"

設定完成後，所有使用者打開網頁時，即使不手動輸入，也會自動套用這組金鑰。

☁️ 部署到 Streamlit Community Cloud

把專案推上 GitHub（記得 .gitignore 已排除 secrets.toml 與 chroma_db/）
前往 share.streamlit.io，用 GitHub 帳號登入
點擊 New app，選擇你的 Repository、分支 main，主檔案指定為 streamlit_app.py
若想讓所有訪客共用同一組 Supabase（方式②），到 Settings → Secrets 貼上：

[supabase]
url = "你的 Project URL"
key = "你的 anon public key"

若想讓每位訪客自行輸入自己的金鑰（方式①），這裡可以完全留空，不影響部署。

點擊 Deploy，等待幾分鐘即可取得公開網址

🧠 8 種 RAG 檢索策略說明
| 策略 | 適合情境 |
|---|---|
| 基礎相似度檢索 | 一般用途，速度快 |
| MMR 多樣性檢索 | 避免檢索結果內容重複 |
| 混合檢索 | 兼顧語意與關鍵字精確比對 |
| 多查詢檢索 | 問題模糊或角度單一時，自動改寫多個問句擴大檢索範圍 |
| 上下文壓縮檢索 | 長文件精簡摘要，減少無關內容干擾 |
| 父子文件檢索 | 需要保留片段前後脈絡時 |
| 自我查詢檢索 | 未來擴充 metadata 篩選功能的基礎架構 |
| 重排序檢索 | 對檢索結果做二次相關性排序，提升精準度 |

⚠️ 已知限制

Streamlit Cloud 免費方案的檔案系統為暫時性，App 重啟後 ChromaDB 索引會被清空（需重新上傳文件）
首次啟動需下載嵌入模型檔案，約需 1-2 分鐘
策略 4（多查詢）與策略 5（上下文壓縮）會額外消耗 Groq API 呼叫次數
若使用方式①（UI 輸入 Supabase 金鑰），關閉分頁或換裝置後需重新輸入一次，因為金鑰只存在當次 Session


這份版本最核心的更新是把 Supabase 章節拆成「① UI 輸入」與「② `secrets.toml`」兩條並列路徑，讓讀者一看就知道自己該選哪一種 — 個人使用選①最省事，團隊共用資料庫選②比較合理。
